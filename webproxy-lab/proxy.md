# 프록시(Proxy)란 무엇인가

프록시는 말 그대로 "대리인"입니다.  
네트워크에서 프록시 서버는 **클라이언트(브라우저, 앱)** 와 **실제 서버** 사이에 끼어들어, 요청과 응답을 대신 전달해 주는 중간 서버입니다.

초심자 관점에서 아주 단순하게 생각하면 다음과 같습니다.

- 원래 방식:
  클라이언트가 서버에 직접 요청함
- 프록시 사용 방식:
  클라이언트가 프록시에 요청함
  프록시가 대신 서버에 요청함
  서버의 응답을 프록시가 다시 클라이언트에게 전달함

예를 들어 브라우저가 어떤 웹페이지를 보고 싶다고 합시다.

1. 브라우저가 프록시 서버에게 "이 주소의 페이지를 가져와 줘"라고 요청합니다.
2. 프록시 서버가 실제 웹 서버에 접속합니다.
3. 실제 웹 서버가 HTML, 이미지, 파일 등을 프록시에게 보냅니다.
4. 프록시는 그 응답을 브라우저에게 다시 전달합니다.

즉, **클라이언트와 서버가 직접 통신하지 않고, 프록시가 중간에서 통신을 대신 수행**합니다.

---

# 왜 프록시를 사용할까

프록시는 단순히 "중간에 하나 더 있는 서버"가 아닙니다.  
중간에 있기 때문에, 요청과 응답을 보면서 여러 가지 중요한 일을 할 수 있습니다.

대표적인 이유는 다음과 같습니다.

## 1. 캐싱(Cache)

프록시는 이전에 받아온 응답을 저장해 둘 수 있습니다.

예를 들어 어떤 학생 100명이 같은 이미지 파일을 요청한다고 합시다.

- 프록시가 없으면:
  학생 100명의 요청이 모두 원본 서버까지 갑니다.
- 프록시가 있으면:
  첫 번째 요청만 원본 서버로 가고,
  나머지 요청은 프록시가 저장해 둔 결과를 바로 돌려줄 수 있습니다.

이렇게 하면 다음과 같은 장점이 생깁니다.

- 응답 속도가 빨라집니다.
- 원본 서버의 부담이 줄어듭니다.
- 네트워크 트래픽이 줄어듭니다.

이번 `proxy.c` 과제도 바로 이 캐시 기능을 핵심으로 다룹니다.

## 2. 보안과 통제

프록시는 중간에서 요청을 통과시키거나 막을 수 있습니다.

예를 들면:

- 특정 사이트 접속 차단
- 특정 파일 형식 다운로드 차단
- 악성 URL 차단
- 내부 사용자 로그 기록

회사, 학교, 기관망에서 프록시를 두는 이유 중 하나가 이것입니다.

## 3. 익명성 또는 정보 은닉

직접 서버에 접속하면 서버는 클라이언트 정보를 바로 보게 됩니다.  
하지만 프록시를 거치면 서버 입장에서는 먼저 프록시와 통신하게 됩니다.

물론 실제 익명성은 프록시 종류와 설정에 따라 크게 달라지지만, 기본 아이디어는 이렇습니다.

- 서버가 직접 클라이언트와 통신하지 않게 함
- 일부 헤더나 정보를 바꾸거나 숨길 수 있음

## 4. 부하 분산 또는 구조 분리

프록시는 요청을 적절한 서버로 나누어 보낼 수 있습니다.

예를 들어:

- 이미지 요청은 이미지 서버로
- API 요청은 API 서버로
- 정적 파일은 CDN이나 캐시 서버로

이런 방식은 대형 서비스에서 매우 중요합니다.

## 5. 프로토콜 처리 단순화

프록시는 서로 다른 네트워크 환경 사이를 연결하는 완충지대 역할도 할 수 있습니다.

예를 들면:

- 클라이언트는 HTTP/1.1을 사용
- 프록시는 HTTP/1.0으로 변환해서 뒤쪽 서버에 전달

이번 과제에서도 브라우저가 `HTTP/1.1`로 보낸 요청을 프록시가 읽고, 서버에는 `HTTP/1.0` 형식으로 다시 보내는 작업을 합니다.

# 프록시의 핵심 개념

프록시를 이해하려면 다음 용어를 구분하는 것이 좋습니다.

## 1. Forward Proxy

클라이언트 앞에 있는 프록시입니다.

- 브라우저/사용자 쪽에 가까움
- 사용자가 프록시를 통해 인터넷에 나감
- 학교/회사/과제에서 자주 보는 형태

이번 `proxy.c`는 이 형태입니다.

브라우저가 다음처럼 요청합니다.

```http
GET http://localhost:15213/home.html HTTP/1.1
```

이 요청은 "프록시야, `localhost:15213` 서버의 `/home.html`을 대신 가져와 줘"라는 뜻입니다.

## 2. Reverse Proxy

서버 앞에 있는 프록시입니다.

- 사용자 입장에서는 실제 서버처럼 보임
- 뒤쪽에 여러 서버를 숨길 수 있음
- 로드 밸런싱, SSL 종료, 캐시, 보안에 많이 사용됨

예를 들어 Nginx가 앞에 있고, 뒤쪽에 여러 앱 서버가 있는 구조가 대표적입니다.

중요한 점은, **이번 과제의 프록시는 리버스 프록시가 아니라 포워드 프록시**라는 것입니다.

## 3. Transparent Proxy

사용자가 프록시를 직접 인식하지 못하는 형태입니다.

- 네트워크 장비나 정책이 자동으로 프록시를 거치게 함
- 기업망, 공공망 등에서 사용되기도 함

## 4. Caching Proxy

응답을 저장해 두었다가 재사용하는 프록시입니다.

- 속도 향상
- 네트워크 절약
- 서버 부하 감소

이번 과제의 `proxy.c`는 바로 이 기능을 포함합니다.

# 프록시의 장점

아래는 프록시를 사용하는 대표적인 장점들입니다.

## 1. 속도 향상

캐시된 응답을 재사용하면 매번 원본 서버까지 갈 필요가 없습니다.

특히 다음 상황에서 효과가 큽니다.

- 같은 파일을 여러 사용자가 반복 요청할 때
- 이미지, CSS, JS, 문서처럼 자주 바뀌지 않는 콘텐츠일 때

## 2. 서버 부하 감소

모든 요청이 원본 서버까지 도달하지 않게 할 수 있습니다.

예를 들어 인기 있는 파일 하나를 1만 명이 요청해도:

- 원본 서버는 1번만 응답하고
- 나머지는 프록시가 캐시에서 처리할 수 있습니다

## 3. 네트워크 효율 증가

외부 회선으로 나가는 트래픽을 줄일 수 있습니다.

학교나 회사처럼 많은 사용자가 같은 외부 자원을 반복 요청하는 환경에서 특히 유리합니다.

## 4. 정책 적용 가능

프록시가 중간에 있기 때문에 다음 같은 정책을 적용할 수 있습니다.

- 특정 도메인 차단
- 특정 메서드 차단
- 헤더 수정
- 접속 기록 저장
- 요청 크기 제한

## 5. 보안 완충 역할

클라이언트가 서버와 직접 연결되지 않으므로, 어느 정도 보호 계층을 둘 수 있습니다.

예를 들면:

- 직접 서버 주소를 숨길 수 있음
- 악성 요청을 사전에 필터링 가능
- 내부망 구조를 감출 수 있음

## 6. 구조 유연성 증가

서비스가 커질수록 프록시는 매우 중요해집니다.

- API 서버를 여러 대로 분산
- 정적 파일을 별도 서버로 분리
- 장애 서버 우회
- 로깅/모니터링 일원화

즉, 프록시는 단순한 중계기가 아니라 **트래픽 제어 지점**이 됩니다.

# 프록시의 단점

프록시가 항상 좋은 것만은 아닙니다. 중간 단계가 하나 늘어나는 것이기 때문에 분명한 단점도 있습니다.

## 1. 구조가 복잡해진다

프록시가 없으면:

- 클라이언트 -> 서버

프록시가 있으면:

- 클라이언트 -> 프록시 -> 서버

구성 요소가 하나 늘어나므로:

- 디버깅이 어려워지고
- 장애 원인 추적이 복잡해지고
- 로그도 여러 군데 봐야 합니다

## 2. 프록시 자체가 병목이 될 수 있다

모든 요청이 프록시를 지나므로 프록시 성능이 부족하면 전체 서비스가 느려질 수 있습니다.

즉:

- 서버가 빨라도
- 프록시가 느리면
- 전체 응답이 느려집니다

## 3. 프록시 장애가 전체 장애로 이어질 수 있다

프록시가 단일 지점(Single Point of Failure)이 되면 위험합니다.

- 프록시가 죽으면 뒤쪽 서버가 살아 있어도 접근이 안 될 수 있음
- 따라서 실무에서는 이중화가 중요함

## 4. 캐시 일관성 문제가 생길 수 있다

캐시의 가장 큰 약점 중 하나입니다.

예를 들어 원본 서버의 파일이 바뀌었는데 프록시가 예전 내용을 계속 들고 있으면:

- 사용자에게 오래된 정보가 전달될 수 있습니다

이번 과제는 교육용 단순 캐시이므로,

- HTTP의 복잡한 캐시 제어 헤더를 모두 해석하지 않고
- 단순히 "작은 객체면 메모리에 저장해서 재사용"하는 방식입니다

실무에서는 훨씬 더 복잡한 정책이 필요합니다.

## 5. 보안/개인정보 문제가 생길 수 있다

프록시는 요청을 중간에서 모두 볼 수 있습니다.

즉:

- 사용자의 접속 정보
- 요청 헤더
- 경우에 따라 민감한 정보

를 다룰 수 있기 때문에, 잘못 운영하면 개인정보 문제나 보안 문제가 생길 수 있습니다.

## 6. 일부 서비스와 충돌할 수 있다

현대 웹 서비스는 단순한 HTTP GET만 쓰지 않습니다.

예를 들면:

- WebSocket
- HTTP/2, HTTP/3
- 긴 연결(keep-alive)
- 인증/세션/쿠키 민감 서비스
- 스트리밍 서비스

프록시가 이런 기능을 잘 지원하지 않으면 오히려 문제가 됩니다.

이번 `proxy.c`도 교육용이므로 `GET` 중심, `HTTP/1.0`, `Connection: close` 기준으로 동작합니다.  
즉, 실제 모든 웹서비스를 완벽하게 프록시할 수 있는 범용 제품은 아닙니다.

# 프록시를 사용하는 이유를 한 문장으로 정리하면

프록시는 **중간에서 요청과 응답을 통제할 수 있기 때문에**, 속도 향상, 캐싱, 보안, 정책 적용, 로깅, 구조 분리 같은 많은 이점을 제공합니다.

즉, 프록시를 쓰는 본질적인 이유는 다음과 같습니다.

- 그냥 전달만 하려는 것이 아니라
- **중간에서 무언가를 하기 위해서** 사용합니다

그 "무언가"는 보통 다음 중 하나입니다.

- 더 빠르게 만들기
- 더 안전하게 만들기
- 더 많이 통제하기
- 더 쉽게 확장하기

# 프록시를 사용해야 하는 경우

다음 상황에서는 프록시가 매우 유용합니다.

## 1. 같은 콘텐츠가 반복 요청되는 경우

예:

- 학교 공지 이미지
- 공통 JS/CSS 파일
- 자주 보는 문서/파일

이럴 때는 캐시 프록시가 큰 효과를 냅니다.

## 2. 외부 인터넷 접속을 통제해야 하는 경우

예:

- 회사에서 특정 사이트 차단
- 학교에서 비인가 서비스 차단
- 보안 정책에 따라 특정 요청 검사

프록시가 정책 집행 지점이 됩니다.

## 3. 로그와 모니터링이 중요한 경우

모든 요청이 프록시를 지나므로:

- 누가 무엇을 요청했는지
- 어떤 서버가 느린지
- 어떤 파일이 자주 요청되는지

를 중앙에서 보기 쉬워집니다.

## 4. 원본 서버 부하를 줄여야 하는 경우

트래픽이 많은 서비스는 프록시나 캐시 계층이 거의 필수입니다.

프록시는 뒤쪽 서버들을 보호하는 역할도 합니다.

## 5. 네트워크 구조를 숨기거나 분리해야 하는 경우

사용자에게 내부 서버 구조를 그대로 노출하고 싶지 않을 때 유용합니다.

실무에서는 리버스 프록시가 이 용도로 많이 쓰입니다.

## 6. 교육용/학습용으로 네트워크 흐름을 배우는 경우

이번 과제처럼 프록시는 다음 내용을 한 번에 공부하기 좋은 주제입니다.

- 소켓 프로그래밍
- HTTP 요청/응답 구조
- 동시성
- 캐시
- 동기화
- 서버 구조

즉, 시스템 프로그래밍 학습용으로도 매우 좋은 예제입니다.

---

# 프록시를 사용하면 안 되거나 주의해야 하는 경우

프록시를 무조건 넣는다고 좋은 것은 아닙니다. 아래 상황에서는 오히려 역효과가 날 수 있습니다.

## 1. 단순하고 작은 시스템

사용자도 적고, 서버도 하나고, 캐시 이점도 거의 없다면 프록시는 과할 수 있습니다.

- 구조만 복잡해짐
- 운영 포인트만 늘어남
- 장애 원인만 하나 더 생김

이럴 때는 직접 연결이 더 단순하고 좋습니다.

## 2. 실시간성이 매우 중요한 경우

프록시가 중간에 있으면 아주 작은 지연이라도 추가될 수 있습니다.

예:

- 초저지연 게임
- 고빈도 거래
- 민감한 실시간 제어 시스템

이 경우 프록시 계층이 부담이 될 수 있습니다.

## 3. 프록시가 프로토콜을 제대로 지원하지 못하는 경우

프록시가 HTTP/1.0만 다루는데, 실제 서비스는 다음을 요구한다면 문제가 생깁니다.

- HTTP/2
- HTTP/3
- WebSocket
- 서버 푸시
- 지속 연결 기반 통신

지원하지 못하는 프록시는 오히려 장애 요인이 됩니다.

## 4. 캐싱하면 안 되는 민감 데이터가 많은 경우

예:

- 개인 정보
- 금융 정보
- 일회성 인증 응답
- 매우 자주 바뀌는 데이터

이런 내용은 잘못 캐시하면 심각한 문제를 만들 수 있습니다.

## 5. 운영 역량이 부족한 경우

프록시는 넣는 것보다 운영이 더 어렵습니다.

- 로그 관리
- 장애 대응
- 캐시 정책
- 보안 설정
- 성능 튜닝

이 준비가 안 된 상태에서 프록시를 넣으면 얻는 것보다 잃는 것이 더 많을 수 있습니다.

## 6. 병목이 프록시 쪽으로 집중되는 경우

요청이 매우 많을 때 프록시가 충분히 강하지 않으면:

- CPU 병목
- 메모리 병목
- 소켓 고갈
- 쓰레드 과다 생성

같은 문제가 발생할 수 있습니다.

즉, 프록시는 "만능 가속기"가 아니라 **잘 설계해야 효과가 나는 중간 계층**입니다.

---

# 프록시를 이해할 때 초심자가 꼭 기억해야 할 포인트

초심자 입장에서 가장 중요한 핵심만 뽑으면 다음과 같습니다.

## 1. 프록시는 중간 서버다

클라이언트와 서버 사이에 위치해서 요청과 응답을 대신 처리합니다.

## 2. 중간에 있기 때문에 많은 일을 할 수 있다

- 캐시
- 차단
- 기록
- 수정
- 분산

이 모든 것이 "중간에 있기 때문에" 가능합니다.

## 3. 하지만 중간에 있기 때문에 책임도 커진다

- 느리면 전체가 느려지고
- 죽으면 전체가 멈추고
- 잘못 캐시하면 잘못된 데이터가 나갑니다

## 4. 프록시는 단순 전달기가 아니다

좋은 프록시는 요청을 "그대로 전달"하는 것이 아니라,
필요에 따라 요청을 "다듬고 관리"합니다.

이번 `proxy.c`도 실제로 다음 일을 합니다.

- URI를 파싱함
- 헤더를 재구성함
- HTTP/1.0으로 바꿈
- 응답을 캐시에 저장함
- 같은 요청이면 서버로 가지 않고 캐시에서 반환함

즉, 프록시는 생각보다 적극적으로 개입하는 프로그램입니다.

# 이번 과제에서의 `proxy.c`와 `tiny.c`의 관계

이제 이번 프로젝트로 내려와서, `proxy.c`와 `tiny.c`가 어떤 관계인지 정리해 보겠습니다.

결론부터 말하면:

- `tiny.c`는 **원본 웹 서버(origin server)** 역할
- `proxy.c`는 **그 앞에 놓인 포워드 프록시** 역할

즉, `tiny.c`가 실제 파일을 들고 있는 서버이고, `proxy.c`는 그 서버에 대신 요청을 보내는 중간자입니다.

---

# 각 파일의 역할

## `tiny.c`의 역할

`tiny.c`는 아주 작은 HTTP 서버입니다.

할 수 있는 일은 다음과 같습니다.

- 클라이언트 요청을 받음
- 정적 파일을 읽어 응답함
- CGI 프로그램을 실행해 동적 응답도 가능
- `GET` 요청 위주로 동작

즉, Tiny는 **"실제 콘텐츠를 제공하는 서버"** 입니다.

예를 들어 Tiny는 이런 파일을 가지고 있습니다.

- `home.html`
- `godzilla.jpg`
- `godzilla.gif`
- `cgi-bin/adder`

브라우저가 Tiny에 직접 접속하면 Tiny가 이 파일들을 읽어 응답합니다.

## `proxy.c`의 역할

`proxy.c`는 브라우저가 직접 원본 서버(Tiny)에 접속하지 않도록 중간에서 대신 처리하는 서버입니다.

할 수 있는 일은 다음과 같습니다.

- 브라우저의 HTTP 요청을 읽음
- URI에서 목적지 호스트/포트/경로를 파싱함
- 필요한 헤더를 다시 구성함
- Tiny 같은 원본 서버에 새 연결을 맺음
- 서버 응답을 받아 클라이언트에 전달함
- 응답이 작으면 캐시에 저장함
- 같은 요청이 오면 캐시에서 바로 응답함

즉, `proxy.c`는 **"대신 요청하고, 대신 응답하고, 필요하면 저장까지 하는 서버"** 입니다.

# 실행 흐름으로 보는 `proxy.c`와 `tiny.c`

브라우저가 프록시를 사용해 Tiny의 `home.html`을 요청한다고 가정해 보겠습니다.

예시 요청:

```bash
curl -v --proxy http://localhost:15214 http://localhost:15213/home.html
```

이 명령의 의미는 다음과 같습니다.

- 실제로 가져오고 싶은 파일은 `http://localhost:15213/home.html`
- 하지만 직접 Tiny에 가지 않고
- `localhost:15214` 에서 실행 중인 프록시를 통해 가져오겠다는 뜻입니다

이때 흐름은 다음과 같습니다.

## 1. 클라이언트가 `proxy.c`에 요청

클라이언트는 프록시에게 이런 형태로 요청합니다.

```http
GET http://localhost:15213/home.html HTTP/1.1
Host: localhost:15213
...
```

중요한 점은 요청줄에 전체 URL이 들어간다는 것입니다.  
이것이 프록시 요청에서 자주 보이는 형태입니다.

## 2. `proxy.c`가 요청을 읽고 해석

`proxy.c`는 다음 정보를 뽑아냅니다.

- hostname: `localhost`
- port: `15213`
- path: `/home.html`

그리고 브라우저 헤더를 적절히 정리합니다.

- `Host` 헤더 유지 또는 재구성
- `User-Agent`는 지정된 값으로 설정
- `Connection: close`
- `Proxy-Connection: close`
- 나머지 헤더는 필요한 범위에서 그대로 전달

## 3. `proxy.c`가 `tiny.c`에 새 연결 생성

이제 프록시는 `Open_clientfd(hostname, port)` 같은 흐름으로 Tiny에 접속합니다.

즉:

- 클라이언트와 프록시 사이 연결 하나
- 프록시와 Tiny 사이 연결 하나

총 두 개의 연결이 생깁니다.

## 4. `proxy.c`가 Tiny에게 새 요청 전송

프록시는 Tiny에게 이런 식으로 요청을 다시 만듭니다.

```http
GET /home.html HTTP/1.0
Host: localhost:15213
User-Agent: ...
Connection: close
Proxy-Connection: close
```

브라우저가 보낸 요청과 완전히 같지는 않습니다.

차이점:

- 전체 URL 대신 경로(`/home.html`)만 보냄
- HTTP/1.1 대신 HTTP/1.0으로 바꿔 보냄
- 필요한 헤더를 다시 정리함

## 5. `tiny.c`가 요청 처리

Tiny는 프록시가 보낸 요청을 마치 일반 클라이언트 요청처럼 처리합니다.

`tiny.c`는:

- 요청줄을 읽고
- `GET`인지 확인하고
- URI를 파일 경로로 바꾸고
- 해당 파일을 읽어 응답합니다

예를 들어 `/home.html`이면 Tiny는 실제 파일을 찾아 읽습니다.

## 6. `tiny.c`의 응답이 `proxy.c`로 돌아옴

Tiny는 HTML 파일 내용을 프록시에게 보냅니다.

이 응답에는 다음이 포함될 수 있습니다.

- 상태줄
- 응답 헤더
- 빈 줄
- 본문 데이터(HTML, 이미지, 바이너리 등)

## 7. `proxy.c`가 응답을 클라이언트에게 전달

프록시는 Tiny에게서 받은 바이트를 그대로 클라이언트에 써 줍니다.

이때 중요한 점은:

- 텍스트만 처리하는 것이 아니라
- 이미지나 바이너리도 그대로 전달해야 한다는 것입니다

그래서 `proxy.c`는 응답을 줄 단위보다 **바이트 스트림 단위**로 읽고 쓰는 방식이 중요합니다.

## 8. `proxy.c`가 응답을 캐시에 저장

응답 크기가 `MAX_OBJECT_SIZE` 이하라면,
`proxy.c`는 응답을 메모리에 저장합니다.

그러면 같은 요청이 다시 왔을 때:

- Tiny에 다시 가지 않고
- 프록시가 메모리에서 바로 응답할 수 있습니다

이것이 이번 과제의 캐시 기능입니다.

# `proxy.c`와 `tiny.c`를 그림처럼 이해하기

다음 구조로 생각하면 쉽습니다.

```text
브라우저(curl)
    |
    | 1. GET http://localhost:15213/home.html HTTP/1.1
    v
proxy.c
    |
    | 2. GET /home.html HTTP/1.0
    v
tiny.c
    |
    | 3. home.html 파일 읽기
    v
proxy.c
    |
    | 4. 응답 전달 + 필요하면 캐시 저장
    v
브라우저(curl)
```

두 번째 요청부터는 캐시 hit가 나면 이렇게 됩니다.

```text
브라우저(curl)
    |
    | GET http://localhost:15213/home.html HTTP/1.1
    v
proxy.c
    |
    | 캐시에서 발견
    | Tiny에 가지 않음
    v
브라우저(curl)
```

# 이번 `proxy.c`에서 실제로 사용된 핵심 개념

이번 구현을 기준으로 보면 `proxy.c`는 다음 개념들을 한꺼번에 보여 줍니다.

## 1. 서버

`proxy.c`는 브라우저 연결을 받는 서버입니다.

- `Open_listenfd`로 포트를 엽니다
- `Accept`로 클라이언트 연결을 받습니다

즉, 프록시는 앞쪽에서는 서버입니다.

## 2. 클라이언트

동시에 `proxy.c`는 Tiny에게 접속하는 클라이언트이기도 합니다.

- `open_clientfd`로 원본 서버에 연결합니다

즉, 프록시는 뒤쪽에서는 클라이언트입니다.

이 점이 초심자에게 아주 중요합니다.  
프록시는 **앞에서는 서버이고, 뒤에서는 클라이언트** 입니다.

## 3. HTTP 요청 파싱

프록시는 요청줄을 보고:

- 메서드
- URI
- 버전

을 해석합니다.

그리고 URI에서:

- 호스트
- 포트
- 경로

를 뽑아냅니다.

## 4. 헤더 재작성

프록시는 브라우저 요청을 그대로 복사만 하지 않습니다.

다음을 직접 제어합니다.

- `Host`
- `User-Agent`
- `Connection`
- `Proxy-Connection`

이것이 프록시가 중간에서 "적극적으로 동작"한다는 증거입니다.

## 5. 동시성

`proxy.c`는 연결마다 스레드를 만들어 동시에 여러 요청을 처리합니다.

이것이 필요한 이유는 간단합니다.

- 어떤 서버는 응답이 매우 느릴 수 있음
- 한 요청 때문에 다른 요청이 전부 막히면 안 됨

그래서 과제에서도 concurrent proxy를 요구합니다.

## 6. 캐시

`proxy.c`는 작은 응답을 메모리에 저장합니다.

이 캐시의 목적은:

- 속도 향상
- 서버 부하 감소

입니다.

## 7. 동기화

여러 스레드가 동시에 캐시에 접근하므로 보호가 필요합니다.

이번 구현에서는:

- 여러 reader는 동시에 읽을 수 있게 하고
- writer는 배타적으로 접근하도록 구성했습니다

이것이 readers-writers 문제와 연결됩니다.

# 초심자가 특히 헷갈리기 쉬운 부분

## 1. 프록시는 단순 복사기가 아니다

요청을 그대로 복사해서 전달하는 프로그램이라고 생각하기 쉽지만,
실제로는 다음을 수행합니다.

- 파싱
- 재구성
- 정책 반영
- 캐시
- 동기화

즉, 꽤 똑똑한 중간 서버입니다.

## 2. `tiny.c`는 프록시가 아니다

`tiny.c`는 원본 서버입니다.

프록시처럼 중간에 서서 대신 전달하는 프로그램이 아니라,
실제로 파일을 제공하는 서버입니다.

## 3. `proxy.c`는 브라우저와 Tiny 둘 다 상대한다

`proxy.c`는 하나의 연결만 보는 프로그램이 아닙니다.

- 브라우저와 대화하는 연결
- Tiny와 대화하는 연결

두 방향을 모두 관리합니다.

## 4. 캐시는 "응답 전체"를 저장하는 것

이번 과제에서는 단순히 파일 이름만 저장하는 것이 아니라,
서버가 보낸 **응답 바이트 전체**를 저장해 두고 다시 보내는 방식으로 이해하면 쉽습니다.

# 마지막 정리

프록시는 클라이언트와 서버 사이에서 요청과 응답을 대신 처리하는 중간 서버입니다.  
사용하는 이유는 단순합니다. 중간에 있기 때문에 **속도 향상, 캐싱, 보안, 제어, 로깅, 구조 분리** 같은 일을 할 수 있기 때문입니다.

하지만 프록시는 만능이 아닙니다.

- 구조를 복잡하게 만들 수 있고
- 병목이 될 수 있고
- 잘못 캐시하면 오래된 정보를 줄 수 있고
- 지원하지 않는 프로토콜에서는 오히려 문제가 됩니다

이번 프로젝트에서:

- `tiny.c`는 실제 콘텐츠를 제공하는 원본 서버
- `proxy.c`는 그 앞에서 요청을 대신 전달하고 캐시하는 포워드 프록시

입니다.

즉, 이 과제는 단순히 프록시 하나를 구현하는 것이 아니라,
다음 개념을 한 번에 배우는 연습이라고 볼 수 있습니다.

- HTTP
- 소켓
- 서버와 클라이언트 역할
- 멀티스레딩
- 캐시
- 동기화

이 문서를 읽은 뒤에는 `proxy.c`를 볼 때 "중간에서 요청을 받아, 다시 요청을 만들고, 응답을 전달하고, 저장하는 프로그램"이라는 큰 그림부터 떠올리면 훨씬 이해가 쉬워집니다.
